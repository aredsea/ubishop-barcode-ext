# 주문 전표 일괄 자동화 설계 (2026-07-20)

주문 전표(`/jun/orderitem/orderItemList.do`)에 세 기능을 추가한다.

| # | 기능 | 요약 |
|---|---|---|
| A | 일괄 수정 | 체크한 건들의 주문비고/발주비고를 한 번에 교체 |
| B | 수정 팝업 | [수정] 클릭 시 화면 이동 대신 플로팅 창에서 편집 |
| C | 본사확인 → 입고완료 | 체크 → 본사확인 → 재고 배정까지 버튼 하나로 |

배포 채널은 **SHELL**(`skin.js`, `manifest.json`, `popup/*`)이라 적용에 크롬 재시작이 필요하다.

---

## 1. 라이브 실측 사실 (2026-07-20, 운영 ERP)

추측이 아니라 실제 화면에서 확인한 계약이다. 이 값들이 바뀌면 설계가 깨진다.

### 목록 페이지

| 항목 | 값 |
|---|---|
| 체크박스 | `input[name=idx]`, value = orderSeq (이 페이지는 단일값, 콤마 없음) |
| 전체선택 | `checkAll(...)` |
| 상태 열 | 헤더 `상태` (관측 시 인덱스 12 — **하드코딩 금지, 헤더에서 찾을 것**) |
| 수정 버튼 | `<a href="javascript:modify('<master>','<seq>')">` → `location.href` 전체 이동 |
| 취소 버튼 | `<a href="javascript:del('<...>')">` |
| 상태 링크 | `<a href="...currentSetting(master, orderSeq, barcode, shop, client, orderDate)">` — **주문완료 행에는 이 링크가 없다** |

### 상단 일괄 버튼 — `standby`

라벨이 `<img>`라 텍스트 검색으로는 찾을 수 없다. 둘 다 `TD.left` 안에 있다.

```
standby(form1, form3, 'OS-', 'O--')   → [본사확인]      주문완료 → 본사확인
standby(form1, form3, 'O--', 'OS-')   → [본사확인취소]  본사확인 → 주문완료
```

`standby`는 체크박스를 읽고 `form3.action`을 세팅해 submit 한다(643자). 실행하면 **페이지가 이동하고 검색조건이 날아간다** — 저장소의 기존 미해결 항목 "상단 일괄 standby 검색조건 미전달"과 같은 함수다.

### 상태 코드

`O--` 주문완료 · `OC-` 주문취소 · `OS-` 본사확인 · `B--` 발주완료 · `I--` 입고완료 · `T--` 출고완료 · `TS-` 출고확인(매장재고) · `TE-` 출고오확인 · `S--` 판매완료

### 수정 폼 (`orderItemModifyForm.do`)

- `form1` → `POST /jun/orderitem/orderItemModify.do`
- **`textarea[shopRemark]` = 주문비고**, **`textarea[remark]` = 발주비고**
- 그 외 필드: `weight` `doc`(hidden) `diaColor`(hidden) `clarity`(hidden) `surface`(hidden) `k` `color` `itemSize` `orderQty` `orderPrice` `orderPrice2` `exdelivedyear/month/day`
- **인코딩 UTF-8** (UTF-8 디코드 시 U+FFFD 0개)

### 배정 팝업 (`orderItemPopCurrentSettingModifyForm.do`)

- **인코딩 EUC-KR** — UTF-8로 읽으면 U+FFFD 56개, EUC-KR로 읽으면 0개
- 표 4열: `No | 입고일자 | 바코드 | 선택`
- 각 행 `<a href="javascript:setCurrent('<바코드>')">`
- `setCurrent`는 인라인 정의. `form1`(POST, action 속성 없음, 이름있는 필드 없음)을 동적으로 채워 제출
- 관측 사례: 후보 **144건**, 전부 입고일자 `2026.05.08` 동일, 페이징 링크 0

> **⚠ 사이트 인코딩이 섞여 있다.** 수정폼은 UTF-8, 배정 팝업은 EUC-KR이다. 읽기는 `postDoc`의 폴백(U+FFFD 20개 초과 시 EUC-KR 재디코드)이 처리하지만, **쓰기는 charset을 틀리면 한글이 깨진 채 저장된다.**

## 3. 아키텍처

### 3.1 쓰기는 네이티브가 한다

저장소의 명시적 원칙은 **"쓰기를 재현하지 말 것"**(`skin.js:788`)이다. 확장은 서버 엔드포인트를 직접 호출하지 않고, 유비샵 자기 코드가 쓰기를 수행하게 한다.

검토한 대안:

- **직접 POST 재현** — 가장 빠르지만 기존 원칙 정면 위반. `setCurrent` 본문을 완전히 읽어내지 못한 상태에서 요청을 재현하면, 서버가 기대하는 상태를 하나라도 빠뜨렸을 때 **재고가 조용히 잘못 배정**된다. 이 프로젝트에서 가장 피해야 할 실패 유형이라 채택하지 않는다.
- **진짜 팝업 창 자동조작** — 기존 가로채기 기계장치를 재사용하지만, N건이면 창이 N번 뜨고 팝업 차단·`opener` 절단 로직과 얽힌다. 폴백으로만 둔다.

### 3.2 두 단계 모두 숨은 iframe에서

`standby`를 현재 페이지에서 부르면 화면이 이동하며 검색조건이 날아가 배치 맥락을 잃는다. 그래서 목록 자체를 화면 밖 iframe에 다시 띄우고 거기서 수행한다. 사용자가 보던 화면은 건드리지 않는다.

```
[본사확인+입고완료] 클릭
 ├ 사전검증 — 체크된 행을 상태별로 분류, "12건 중 9건 대상 / 3건 제외(사유)" 승인
 ├ Phase 1 (iframe, 목록)  대상 orderSeq만 체크 → standby(본사확인) 클릭 → OS- 전환 확인
 ├ Phase 2 (iframe, 팝업)  건별 순차: 팝업 로드 → 첫 행 setCurrent 클릭 → 결과 검증
 │                          └ 실패 시 즉시 중단
 └ 요약 표시 + 사용자 화면 목록 갱신
```

### 3.3 자동화 프레임 가드 (신규 필수 부품)

`manifest.json:90`에서 **`skin.js`는 `all_frames: true`**이고 프레임 가드가 없다(`isTop`을 로그로 찍기만 한다). 방치하면 iframe마다 skin.js 전체가 실행되어:

- 사이드바가 iframe 안에 그려진다
- `bindAssignIntercept`의 capture 클릭 리스너가 **우리의 프로그램적 클릭을 삼킨다**
- `chrome.storage.onChanged`가 모든 컨텍스트에 방송되어(`skin.js:805` 함정) 중복 반응이 생긴다

대책: 자동화 iframe URL에 `ubauto=<nonce>`를 부여하고, `skin.js` 진입부에서 이를 감지하면 **UI 기능을 전부 비활성화**한다(사이드바·클릭 가로채기·매입처창·캐시). 기존 `ubasg` nonce 파라미터와 같은 패턴이다.

### 3.4 팝업 헬퍼 일반화

현재 `fwEnsurePanel()`은 `FW_ID`와 저장키 `ubFwX/Y/W/H`가 하드코딩되어 **인스턴스가 1개뿐**이다. `id`·저장키 접두사를 인자로 받게 일반화한다. CSS `.ub-fw-*`는 공유하고(id 중복주입 가드가 이미 있다), **매입처 창의 동작은 그대로 유지**하며 회귀 테스트로 고정한다.

이 작업의 목적에 직접 필요한 범위로만 손댄다. 무관한 리팩토링은 하지 않는다.

## 4. 기능별 명세

구현은 **A → B → C 순서**로 단계를 나눈다. 위험도가 낮은 것부터 올려 3.3·3.4의 공용 부품(프레임 가드·패널 일반화)을 A에서 먼저 실전 검증하고, 되돌리기 가장 어려운 C를 마지막에 둔다. 각 단계는 독립 커밋이며 그 단계만으로 동작해야 한다.

### 4.1 기능 A — 일괄 수정 (먼저 구현)

- **버튼**: `TD.left` 안, `standby` 앵커 두 개 옆에 [일괄 수정]
- **팝업**: 일반화된 패널. `☐ 주문비고` / `☐ 발주비고` 각각 체크박스 + textarea, [적용] [취소]
- **동작**: 체크한 칸만 적용. 덮어쓰기(교체). 체크 안 한 칸은 건드리지 않는다
- **건별 처리**
  1. 행의 `modify('<master>','<seq>')` 인자를 읽는다
  2. `orderItemModifyForm.do` GET
  3. **`form1` 전체를 그대로 직렬화**하고 `shopRemark`/`remark`만 교체
  4. `POST orderItemModify.do`
  5. 응답 문구를 읽지 않고 **수정폼을 다시 GET 해서 해당 textarea 값이 의도한 문자열과 같은지 대조**한다. 비고는 목록 화면에 없으므로 재조회 대상은 목록이 아니라 수정폼이다 (`skin.js:796` — 이 앱에서 응답 문구 스캔은 항상 오탐)
- 순차 처리, **첫 실패에서 중단**
- 완료 후 화면 갱신: 비고는 목록에 표시되지 않으므로 **목록을 건드리지 않는다**. 결과 요약만 표시

> 일부 필드만 보내면 나머지가 지워진다. 반드시 전체 왕복.

### 4.2 기능 B — 수정 팝업

- **가로채기**: 목록의 `a[href*="modify("]` 를 **capture 단계**에서 잡아 `preventDefault` → 패널 열기
- **패널 내용**: 네이티브 수정폼을 **iframe으로 그대로 로드**한다. 폼을 재구성하지 않는 이유는 hidden 필드·select·자체 검증 JS가 많아 재현하면 틀릴 곳이 많기 때문이다
- iframe 안 ERP 헤더·메뉴는 같은 출처이므로 스타일을 주입해 가리고 폼만 보이게 한다
- **저장**: iframe이 결과 페이지로 이동하는 것을 감지 → 패널을 닫고, 그 행만 기존 `fetchStatusCell` + `resyncRow` 경로로 갱신한다. 그 경로가 실패하면 목록 전체를 새로고침한다(조용히 낡은 값을 두지 않는다)
- **게이트 OFF면 가로채지 않는다** — 네이티브 그대로 화면 이동

### 4.3 기능 C — 본사확인 → 입고완료

- **버튼**: `TD.left` 에 [본사확인+입고완료]
- **사전검증 분류**
  - 대상: `O--`(주문완료) — Phase 1부터
  - 대상: `OS-`(본사확인) — Phase 1 건너뛰고 Phase 2부터
  - 제외: `I--` `OC-` `T--` `TS-` `TE-` `S--` `B--`
  - "N건 중 M건 대상 / K건 제외(사유)"를 보여주고 승인받은 뒤 시작
- **Phase 1** (iframe, 목록): 대상만 체크 → `standby` 클릭 → 해당 orderSeq가 `OS-`로 바뀌었는지 확인
- **Phase 2** (iframe, 팝업): 건별 순차
  - 팝업 URL은 기존 `buildAssignPopupUrl` 규약을 따른다(고정 6인자 + `form1`의 `ASG_SEARCH_FIELDS` 24개 복사, `f.elements[n]` 사용 — `f[n]`은 동명 필드에서 조용히 누락된다)
  - 후보 0건이면 실패로 처리하고 중단
  - 첫 행의 `setCurrent` 링크 클릭
  - **검증은 기존 `fetchStatusCell(orderSeq, orderDate)`** 를 쓴다(8초 타임아웃, 화면 필터를 물려받지 않음 — `skin.js:1107`의 이유가 그대로 유효)
- **첫 실패에서 즉시 중단** → "성공 5건 / 6번째에서 중단(사유)" 요약
- 완료 후 화면 갱신: 처리된 orderSeq들을 기존 `fetchStatusCell` + `resyncRow` 경로로 행 단위 갱신하고, 그 경로가 실패하면 목록 전체를 새로고침한다. 중단된 경우에도 **이미 처리된 건까지는 반드시 갱신**한다 — 서버는 바뀌었는데 화면만 옛 상태로 남는 것이 이 서브시스템에서 반복해 당한 실패다(`skin.js:898`·`1142`)

## 5. 옵션 등록

세 기능 모두 `D`(skin.js) + `popup/popup.html` + `popup/popup.js`(기본값·render·리스너·disabled 배열) 3곳에 등록한다.

**게이트는 초기화 시점과 클릭 시점 양쪽에서 확인한다**(`skin.js:927`·`1608` 원칙). 초기화 때만 보면 켠 채로 연 페이지는 끌 수가 없다.

참고: 기존 `ubFactoryInfo`는 `D`에만 있고 popup UI에 노출되지 않아 사실상 끌 수 없다. 이번에 같이 노출한다.

## 6. 테스트 전략

**순수 헬퍼를 DOM 비의존으로 분리**해 node 테스트로 고정한다.

- 상태 분류 / 대상·제외 필터링
- 폼 직렬화 + 비고 교체 (전체 필드 보존 확인)
- 팝업 첫 행 선택 + 후보 0건 처리
- 팝업 헬퍼 일반화 후에도 매입처 창 동작이 동일한지 (회귀)

> ⚠ `tests/orderitem-assign.test.js`는 `function 이름(` 문자열로 소스를 추출한다(21~31행). 대상 함수를 화살표 함수나 메서드로 쓰면 **테스트가 즉사**한다. `function` 선언으로 작성할 것.

**라이브 검증**은 1건짜리 실제 배정으로 시작한다. 되돌리기 어려우므로 사용자 확인 후 진행한다.

| R3 | `setCurrent` 본문 전체 미확인 | 네이티브를 호출하는 설계라 본문을 몰라도 된다. 직접 POST를 채택하지 않은 이유이기도 함 |
| R4 | Phase 1의 `standby`가 iframe 안에서 검색조건 없이 동작할 때의 부작용 | Phase 1 직후 대상 orderSeq의 상태를 재조회로 확인. 불일치면 즉시 중단 |
| R5 | 후보가 페이징되는 경우(관측 사례는 144건 단일 페이지) | 첫 페이지의 첫 행만 사용. 페이징이 나타나면 그때 재검토 |

## 8. 범위 외

- `standby`의 "검색조건 미전달" 자체를 고치는 것 (별건, 기존 미해결 항목)
- 배정 규칙을 FIFO 등으로 바꾸는 것 (현행 관행 보존이 이번 결정)
